# MCP Tools v2.0 — Nuevas tools + Schemas faltantes

Convenciones:
- Standard: JSON Schema Draft 2020-12
- English-only para artefactos técnicos (tool names, JSON keys, enums, schemas)
- `school_id` resuelto por JWT/sesión excepto en tools institucionales (ADM/Analytics)
- Todas las Action con efectos persistentes llevan `idempotency_key`
- Acciones irreversibles llevan `explicit_confirmation: boolean, const: true`
- `additionalProperties: false` en todos los objetos

### Definiciones por tool

**`revoke_guardian_access@v1`**
Bloquea de forma inmediata todos los accesos de un tutor a la plataforma en todos los canales (app, web, WhatsApp) vía actualización de RLS. Acción irreversible vía asistente — el reestablecimiento requiere flujo manual administrativo. Registra motivo y actor en audit log. Bloquea notificaciones salientes hacia ese tutor. No elimina el vínculo tutor-alumno del modelo de datos: solo suspende acceso.

**`search_guardian@v1`**
Busca tutores dentro del tenant por nombre parcial o DNI. Devuelve máximo 10 resultados para evitar exposición masiva. Aplica RLS por `school_id`. Solo roles administrativos. Nunca devuelve datos financieros ni conversaciones.

**`register_consent@v1`**
Registra el consentimiento explícito de un tutor al incorporarse a la plataforma. Almacena versión del documento legal aceptado, timestamp del servidor (no del cliente), IP de origen y opciones específicas marcadas (ej: comunicaciones por WhatsApp, uso de datos con fines pedagógicos). Un tutor puede tener múltiples registros de consentimiento si cambia la versión del documento. Idempotente por `(guardian_id, document_version)`.

**`get_consent_status@v1`**
Verifica si un tutor tiene consentimiento activo y vigente para la versión actual del documento legal. Devuelve estado, fecha de aceptación y opciones elegidas. Usado por el sistema antes de enviar comunicaciones por canales externos (WhatsApp).

**`export_user_data@v1`**
Compila y exporta todos los datos personales de un usuario (tutor o familia) como respuesta al derecho de Acceso ARCO (Ley 25.326). Consulta múltiples tablas: perfil, hijos vinculados, pagos, comunicaciones, consentimientos, conversaciones. Genera un archivo estructurado (JSON/ZIP). El proceso puede ser asincrónico; devuelve un `job_id` para polling si la compilación supera 10 segundos. No expone datos de otros usuarios de la institución.

**`request_data_rectification@v1`**
Genera un ticket formal de rectificación de datos (derecho ARCO-Rectificación) con SLA trazable. No modifica datos directamente — crea una solicitud que debe ser procesada por un administrador. Devuelve `ticket_id` y fecha límite de respuesta según normativa. El asistente no ejecuta la rectificación autónomamente.

**`request_data_deletion@v1`**
Inicia el flujo de eliminación parcial de datos personales (derecho ARCO-Supresión). Genera solicitud con estado `pending_review` que requiere aprobación de admin antes de ejecutar. El admin recibe alerta proactiva. Acción irreversible una vez aprobada — requiere `explicit_confirmation`. No elimina datos requeridos por obligaciones legales/fiscales (facturas, comprobantes).

**`get_reenrollment_status@v1`**
Devuelve el estado de reinscripción de la escuela para un ciclo lectivo dado, segmentado por riesgo (confirmados, pendientes, en riesgo, sin respuesta). Permite al admin identificar familias que requieren seguimiento proactivo. Incluye métricas de conversión del proceso de reinscripción.

**`create_reenrollment_campaign@v1`**
Crea y envía una campaña de reinscripción diferenciada por segmento de riesgo. Genera mensajes personalizados por segmento. Requiere preview antes del envío. Bloquea envío a familias sin opt-in activo. Registra campaña con estado y métricas de respuesta.

**`get_daily_journal@v1`**
Obtiene el diario visual del día para un alumno de nivel inicial o primaria. El diario incluye actividades realizadas, fotos (si la docente las publicó), estado emocional y nota narrativa. Los padres lo ven en modo lectura. Solo disponible para niveles inicial y primaria 1er ciclo.

**`publish_daily_journal@v1`**
Publica el diario del día generado por la docente. Valida que tenga al menos una actividad registrada. Notifica a los padres de los alumnos del grado. Idempotente por `(grade_id, date)` — si ya existe un diario publicado para esa fecha, actualiza en lugar de duplicar.

**`get_teacher_portfolio@v1`**
Obtiene el portfolio pedagógico de un docente para un período dado. Incluye actividades generadas y publicadas, informes trimestrales redactados, observaciones registradas, métricas de engagement (tasas de lectura de comunicados, asistencia tomada en tiempo). Modo interno devuelve métricas completas; el docente solo ve su propio portfolio.

**`generate_teacher_portfolio_pdf@v1`**
Genera un PDF descargable del portfolio docente. Modo `internal` incluye métricas institucionales y es para uso del directivo. Modo `external` es el CV pedagógico del docente (sin métricas de la escuela). Proceso asincrónico si el período es mayor a un trimestre; devuelve `job_id` para polling.

**`activate_event_mode@v1`**
Activa el modo "corresponsal de eventos" para un evento del calendario escolar (acto, salida, competencia). Habilita el canal de publicación de updates en tiempo real para ese evento. Define si las fotos están bloqueadas (eventos con menores en espacio público). Establece la ventana de tiempo activa del modo. Idempotente por `event_id`.

**`publish_event_update@v1`**
Publica un update en tiempo real durante un evento activo. Valida que el modo evento esté activo para el `event_id`. Si `photo_blocked = true` en el evento, bloquea cualquier payload con `photo_url`. Distribuye el update a los destinatarios configurados en el evento. Devuelve `update_id` para el álbum posterior.

**`generate_event_album@v1`**
Genera el álbum cronológico del evento consolidando todos los updates publicados durante el modo evento. Solo puede ejecutarse una vez finalizado el modo evento. Devuelve URL del álbum para compartir con familias. Proceso asincrónico si hay más de 20 fotos; devuelve `job_id`.
